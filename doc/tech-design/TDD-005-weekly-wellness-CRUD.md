# TDD-005 — WeeklyWellness CRUD

## Context
Fourth domain. Introduces weekly wellness surveys linked to athletes.
One survey per athlete per week (Monday to Sunday).
Feeds the WellnessReport aggregate (TDD-008) and CorrelationReport (TDD-009).
Follows the same patterns as TDD-003 (TrainingSession) — sub-resource of athlete,
ownership chain Coach→Athlete→WeeklyWellness.

## Data Model

### Flyway Migration
- `V6__create_weekly_wellness_table.sql`

### Table: weekly_wellness
| Column | Type | Constraints |
|---|---|---|
| id | BIGINT GENERATED BY DEFAULT AS IDENTITY | PK |
| athlete_id | BIGINT | NOT NULL, FK → athlete(id) |
| week_start_date | DATE | NOT NULL |
| perceived_difficulty | INTEGER | NOT NULL, CHECK 1-5 |
| perceived_fatigue | INTEGER | NOT NULL, CHECK 1-5 |
| motivation | INTEGER | NOT NULL, CHECK 1-5 |
| — | — | UNIQUE (athlete_id, week_start_date) |

### Domain Entity
`WeeklyWellness` — pure POJO, Lombok `@Getter @EqualsAndHashCode(exclude="id") @ToString`.
Final fields except `id`. Manual constructor with domain invariants.

### Domain Invariants
- `athleteId` — not null, > 0
- `weekStartDate` — not null, must be a Monday (`DayOfWeek.MONDAY`)
- `perceivedDifficulty`, `perceivedFatigue`, `motivation` — between 1 and 5 inclusive
- Violation → `DomainValidationException`

## Domain Interfaces

### New port (domain/)
```java
public interface WeeklyWellnessRepository {
    WeeklyWellness save(WeeklyWellness wellness);
    Optional<WeeklyWellness> findById(long id);
    List<WeeklyWellness> findByAthleteIdAndPeriod(long athleteId, LocalDate from, LocalDate to);
    void deleteById(long id);
    boolean existsById(long id);
    boolean existsByAthleteIdAndWeekStartDate(long athleteId, LocalDate weekStartDate);
}
```

### New exceptions (domain/exception/)
- `WeeklyWellnessNotFoundException(long id)` → 404
- `WeeklyWellnessAlreadyExistsException(long athleteId, LocalDate weekStartDate)` → 409

### New ErrorCode
- `WELLNESS_ALREADY_EXISTS` — one wellness entry per athlete per week

## Service flows

### Create flow
`WeeklyWellnessRequest`
→ `WeeklyWellnessWebMapper.requestToDomain(request, athleteId)`
→ `WeeklyWellnessService.save(WeeklyWellness)`
→ `wellnessRepository.existsByAthleteIdAndWeekStartDate()` → `WeeklyWellnessAlreadyExistsException` if true
→ `wellnessRepository.save(wellness)` → `WeeklyWellness`
→ `WeeklyWellnessWebMapper.domainToResponse()` → `WeeklyWellnessResponse`

### Update flow
`WeeklyWellnessRequest`
→ `WeeklyWellnessService.update(WeeklyWellness)`
→ `wellnessRepository.findById()` → `WeeklyWellnessNotFoundException` if empty
→ verify `wellness.athleteId == athleteId` → 404 if mismatch
→ `existsByAthleteIdAndWeekStartDate()` if weekStartDate changed → 409 if exists
→ `wellnessRepository.save(wellness)`

### Get by period flow
`from` (required), `to` (optional, defaults to today)
→ validate `from <= to` in controller → 400 if invalid
→ `WeeklyWellnessService.findByAthleteIdAndPeriod(athleteId, from, to)`
→ `wellnessRepository.findByAthleteIdAndPeriod(athleteId, from, to)`
→ `List<WeeklyWellness>` mapped to `List<WeeklyWellnessResponse>`

## Decisions
- `weekStartDate` must be Monday — enforced in domain constructor via `DayOfWeek.MONDAY` check
- Uniqueness `(athlete_id, week_start_date)` enforced at both DB level (UNIQUE constraint) and service level (`existsByAthleteIdAndWeekStartDate`) — defense in depth
- `WeeklyWellnessAlreadyExistsException` → 409 Conflict — client can GET the existing entry and PUT to update it
- `WeeklyWellnessRequest.weekStartDate` annotated with `@PastOrPresent` — future wellness
  entries have no business value; validation at HTTP boundary
- Values 1-5 enforced at three levels: DB CHECK constraint, domain constructor invariant, Bean Validation on DTO
- `athleteId` injected from `@PathVariable`, not in request body — consistent with `TrainingSession`
- `WeeklyWellnessWebMapper.requestToDomain(request, athleteId)` — two parameters, consistent with `TrainingSessionWebMapper`
- Ownership chain: controller validates `Coach→Athlete` via `athleteService.findById()`, service validates `Wellness→Athlete` via `existsById` — consistent with `TrainingSession`, to be refactored in hexagonal phase
- 404 instead of 403 on ownership violation — prevents id enumeration
- `@Transactional(readOnly = true)` at class level, `@Transactional` on write methods
- `GET /v1/athletes/{id}/wellness` uses period filter (`from` required, `to` optional defaults to today) instead of 
  pagination — use case is time-series display and WeeklyReport aggregation, not generic list browsing
- `from <= to` validated at web layer only — query parameter constraint, not a domain invariant nor a DB constraint
- `WeeklyWellnessJpaRepository` derived method : `findAllByAthleteIdAndWeekStartDateBetween(long athleteId, 
LocalDate from, LocalDate to)` — Spring Data generates `WHERE week_start_date >= from AND week_start_date <= to` 
  (inclusive)
